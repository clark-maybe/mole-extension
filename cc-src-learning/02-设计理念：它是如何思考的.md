# 第二章：设计理念——它是如何"思考"的？

如果说第一章让我们对 Claude Code 有了一个整体的认识，那么这一章我们将深入到它的"思维方式"。Claude Code 之所以能够像一个真正的程序员一样帮你写代码、修改 bug、执行复杂的任务，很大程度上得益于它精心设计的几个核心系统。

在这一章中，我们将逐一解析：
- **工具系统**：如何让 AI 具备各种能力（读文件、执行命令等）
- **命令系统**：如何通过自然语言触发特定功能
- **权限系统**：如何在给予 AI 能力和保护用户安全之间取得平衡
- **代理系统**：如何让多个 AI 协同工作
- **技能系统**：如何把常用操作变成可复用的模板
- **插件系统**：如何让第三方扩展这个工具的能力

让我们开始吧。

---

## 2.1 像乐高一样的工具系统（Tool）

### 什么是工具？

在 Claude Code 中，"工具"（Tool）是 AI 与外界交互的桥梁。当你让 Claude Code "帮我读取这个文件"时，它会调用 `FileReadTool`。当你让它"运行这个命令"时，它会调用 `BashTool`。

这就像乐高积木——每块积木都有特定的形状和功能，你可以用它们拼出各种各样的东西。

### 工具的结构

让我们以 `BashTool`（执行命令行工具）为例，看看一个工具是如何定义的：

```typescript
// src/tools/BashTool/prompt.ts（简化）
function getSimplePrompt(): string {
  return [
    'Executes a given bash command and returns its output.',
    '',
    '# Instructions',
    'If your command will create new directories or files, first use this tool to run `ls` to verify the parent directory exists...',
    // ... 更多指令
  ].join('\n')
}
```

每个工具都有两个核心部分：

1. **工具描述（Prompt）**：告诉 AI 这个工具是做什么的、怎么用
2. **执行逻辑**：实际执行操作的代码

### 工具的注册机制

所有的工具都注册在 `src/tools.ts` 中。这是一个典型的"注册表模式"：

```typescript
// src/tools.ts
import { AgentTool } from './tools/AgentTool/AgentTool.js'
import { SkillTool } from './tools/SkillTool/SkillTool.js'
import { BashTool } from './tools/BashTool/BashTool.js'
import { FileEditTool } from './tools/FileEditTool/FileEditTool.js'
// ... 更多导入

// 统一注册
export const tools = [
  AgentTool,
  SkillTool,
  BashTool,
  FileEditTool,
  // ...
]
```

这种设计的好处是：**新增一个工具就像添加一块乐高积木，不需要修改核心代码**。

### 工具的分类

Claude Code 内置了大约 40 种工具，我们可以把它们分成几大类：

| 类别 | 工具示例 | 功能 |
|------|----------|------|
| **文件操作** | FileReadTool, FileWriteTool, FileEditTool | 读取、写入、修改文件 |
| **代码搜索** | GlobTool, GrepTool, LSPTool | 按模式搜索文件、搜索代码内容 |
| **命令执行** | BashTool, PowerShellTool | 执行 shell 命令 |
| **网络请求** | WebFetchTool, WebSearchTool | 获取网页内容、搜索互联网 |
| **代理/团队** | AgentTool, SendMessageTool, TeamCreateTool | 创建子代理、多代理协作 |
| **任务管理** | TaskCreateTool, TaskListTool, TaskUpdateTool | 创建和管理任务 |
| **时间调度** | CronCreateTool, CronDeleteTool, CronListTool | 定时任务管理 |
| **技能** | SkillTool | 执行预定义的技能模板 |
| **MCP 集成** | MCPTool, ListMcpResourcesTool | 与 Model Context Protocol 服务器通信 |

### 工具的"说明书"：Prompt 是怎么写的

这是最有趣的部分。工具的 prompt 写得越好，AI 就越知道怎么正确使用这个工具。

以文件编辑工具为例：

```typescript
// src/tools/FileEditTool/prompt.ts
function getDefaultEditDescription(): string {
  return `Performs exact string replacements in files.

Usage:
- You must use your \`${FILE_READ_TOOL_NAME}\` tool at least once in the conversation before editing.
- When editing text from Read tool output, ensure you preserve the exact indentation...
- ALWAYS prefer editing existing files in the codebase. NEVER write new files unless explicitly required.
- The edit will FAIL if \`old_string\` is not unique in the file...
- Use \`replace_all\` for replacing and renaming strings across the file.`
}
```

这个 prompt 包含了：
1. **功能描述**：这个工具是做什么的
2. **使用前提**：必须先读取文件才能编辑
3. **格式要求**：如何保留缩进、如何指定要替换的文本
4. **常见错误**：什么情况会导致编辑失败
5. **最佳实践**：什么时候该用 `replace_all`

### 工具权限：安全第一

每个工具都有一个权限级别。Claude Code 使用权限系统来决定是否允许 AI 调用某个工具。

```typescript
// 权限级别定义
type PermissionMode =
  | 'default'    // 默认模式，每次都要询问
  | 'plan'       // 计划模式，批量确认
  | 'auto'       // 自动模式，某些操作自动允许
  | 'bypassPermissions'  // 绕过权限检查（需要特殊配置）
```

### 小结：工具系统的设计哲学

工具系统的设计体现了几个核心理念：

1. **单一职责**：每个工具只做一件事，但把那件事做到极致
2. **清晰的接口**：通过 prompt 让 AI 准确地理解每个工具的用法
3. **可扩展性**：新工具只需要实现接口，就能无缝接入
4. **安全可控**：通过权限系统，让用户始终掌握最终决定权

## 2.2 slash 命令：像聊天机器人一样的交互方式

### 什么是 slash 命令？

当你用 Claude Code 时，你会输入 `/commit`、`/review`、`/help` 这样的命令。这些以斜杠开头的命令就是"slash 命令"。

这就像在 Slack 或 Discord 中输入 `/gif` 来搜索 GIF 图一样——slash 命令提供了一种结构化的方式来触发特定功能。

### 命令的实现

slash 命令的注册在 `src/commands.ts` 中。让我看一个例子——`/commit` 命令：

```typescript
// src/commands/commit.ts
export default async function commitHandler(args: { message: string }) {
  // 1. 获取 git 状态
  // 2. 分析变更
  // 3. 生成提交信息
  // 4. 执行 git commit
}
```

命令的核心是一个**处理器函数**（handler），它接收参数并执行相应的逻辑。

### 命令的注册表

在 `commands.ts` 中，所有命令被注册到一个统一的表中：

```typescript
// src/commands.ts（简化）
const commands = [
  { name: 'commit', handler: commit, description: 'Create a git commit' },
  { name: 'review', handler: review, description: 'Code review' },
  { name: 'compact', handler: compact, description: 'Context compression' },
  { name: 'mcp', handler: mcp, description: 'MCP server management' },
  // ... 更多命令
]
```

### 命令的参数解析

slash 命令支持参数。拿 `/review pr#123` 为例：

```typescript
// 命令行输入：/review pr#123
// 会被解析为：
{
  command: 'review',
  args: { prNumber: '123' }
}
```

参数解析使用了一个自定义的 `argumentSubstitution` 系统，支持：
- **位置参数**：`/send "hello" "world"`
- **命名参数**：`/review pr#123`
- **可选参数**：有些参数可以省略

### slash 命令 vs 普通对话

为什么要用 slash 命令，而不是直接说"帮我提交代码"？

| 对话方式 | 例子 | 优点 | 缺点 |
|----------|------|------|------|
| **自然语言** | "帮我提交代码" | 灵活、自然 | 可能被误解 |
| **slash 命令** | `/commit -m "fix bug"` | 结构化、精确 | 需要记忆命令 |

Claude Code 两者的优点兼得：
- 对于复杂或精确的操作，提供 slash 命令
- 对于日常对话，使用自然语言理解

### 命令的分类

Claude Code 提供了约 50 个 slash 命令，我们可以把它们分成几类：

| 类别 | 命令 | 功能 |
|------|------|------|
| **Git 操作** | `/commit`, `/diff`, `/branch` | Git 相关操作 |
| **代码审查** | `/review`, `/compact` | 代码审查和压缩 |
| **搜索** | `/search`, `/grep` | 搜索文件和代码 |
| **MCP** | `/mcp add`, `/mcp list` | MCP 服务器管理 |
| **配置** | `/config`, `/model`, `/theme` | 修改设置 |
| **诊断** | `/doctor`, `/cost`, `/status` | 系统状态和诊断 |
| **会话** | `/resume`, `/clear`, `/share` | 会话管理 |

## 2.3 权限系统：为什么每次都要问我"允许吗"？

### 权限系统的必要性

当你让 Claude Code "删除这个文件夹"时，它会弹出"是否允许删除？"的提示。这是权限系统在起作用。

为什么要这样做？因为 AI 虽然强大，但并不是完美的——它可能会：
- 误解你的意图，删除了错误的文件
- 执行了不安全的操作，如删除系统文件
- 意外修改了不打算改的代码

权限系统就像是一个"安全阀"，确保用户始终掌控 AI 的行为边界。

### 权限的级别

Claude Code 定义了多个权限级别：

```typescript
type PermissionMode =
  | 'default'    // 每次操作前都询问
  | 'plan'       // 在执行前列出所有操作，用户确认
  | 'auto'       // 某些操作自动允许，其他仍需询问
  | 'bypassPermissions'  // 完全信任 AI（危险，仅用于特殊场景）
```

### 权限检查的流程

当 AI 想要执行一个操作时，权限系统会经历以下流程：

```
┌─────────────────────────────────────────────────────────────┐
│                    权限检查流程                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 触发检查                                                 │
│     └─→ AI 调用工具 → 系统拦截                               │
│                                                             │
│  2. 分类判断                                                 │
│     ├─→ 是否在"自动允许"列表中？ → 直接执行                   │
│     ├─→ 是否在"自动拒绝"列表中？ → 拒绝执行                   │
│     └─→ 需要询问用户？ → 进入等待队列                         │
│                                                             │
│  3. 用户响应                                                 │
│     ├─→ 允许 → 执行操作                                     │
│     ├─→ 拒绝 → 返回错误给 AI                                │
│     └─→ 记住选择？ → 更新权限规则                            │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 基于上下文的权限

权限系统还会考虑上下文。比如：

```typescript
// 某些操作在不同场景下有不同权限
const rules = {
  // 在 plan 模式下，某些危险操作被降级
  dangerousInPlan: ['rm -rf', 'git reset --hard'],
  // 但在特定目录下的操作可能更宽松
  projectSpecific: {
    '/tmp': 'auto',  // /tmp 下的操作自动允许
    '~/.config': 'ask'  // 配置目录需要询问
  }
}
```

### 权限规则的可持久化

用户做出的权限决定可以被记住。Claude Code 会把决定存储在 `~/.claude/settings.json` 中：

```json
{
  "permissions": {
    "allow": [
      "BashTool:git status",
      "BashTool:git diff"
    ],
    "deny": [
      "BashTool:rm -rf"
    ]
  }
}
```

这样下次遇到同样的操作时，就不需要再询问了。

### 沙箱（Sandbox）机制

除了用户确认机制，Claude Code 还引入了**沙箱**来限制命令的实际影响范围：

```typescript
// src/tools/BashTool/prompt.ts
function getSimpleSandboxSection(): string {
  return `
## Command sandbox
By default, your command will be run in a sandbox. This sandbox controls
which directories and network hosts commands may access...

The sandbox has the following restrictions:
- Filesystem: only specific directories can be written
- Network: only specific hosts can be accessed
`
}
```

沙箱配置可以非常精细：

```typescript
const sandboxConfig = {
  filesystem: {
    write: {
      allowOnly: ['/tmp/claude-1001', '~/projects'],  // 只允许写入这些目录
      denyWithinAllow: ['~/projects/.git']             // 但禁止写入 .git
    }
  },
  network: {
    allowedHosts: ['api.github.com', 'npm.org'],        // 只允许访问这些网络地址
    deniedHosts: ['internal.corp.local']                // 禁止访问这些地址
  }
}
```

### 小结：安全与效率的平衡

权限系统的设计体现了几个核心原则：

1. **用户主权**：最终决定权始终在用户手中
2. **最小惊讶**：通过规则学习，避免重复确认常见的合理操作
3. **深度防御**：即使权限判断出错，沙箱也能阻止真正的破坏
4. **透明可控**：用户可以随时查看、修改权限规则

## 2.4 子代理与团队协作：一个人不够，那就多来几个

### 为什么需要子代理？

想象你是一个项目经理。当你负责一个小项目时，你可以独自完成所有工作。但当你负责一个大项目时，你会把任务分解，分配给不同的团队成员。

Claude Code 的子代理系统就是基于同样的理念。当 AI 遇到一个复杂任务时，它可以创建子代理来并行处理不同的部分。

### AgentTool：创建子代理

在代码中，子代理是通过 `AgentTool` 来创建的：

```typescript
// 用户说："帮我审查这个 PR，同时更新相关文档"
// AI 会这样响应：
await AgentTool({
  description: "PR code review",
  prompt: "Review PR #123 for code quality, security issues...",
  subagent_type: "code-reviewer"
})

await AgentTool({
  description: "Update documentation",
  prompt: "Update API docs for the changes in PR #123...",
  subagent_type: "documentation-writer"
})
```

两个子代理会**并行运行**，大大提高效率。

### Agent 的类型

Claude Code 预定义了几种专门的 Agent：

```typescript
// src/tools/AgentTool/built-in/exploreAgent.ts
export const EXPLORE_AGENT = {
  agentType: 'explore',
  description: 'File search specialist for exploring codebases',
  // 只允许读操作，不能修改任何文件
  tools: ['GlobTool', 'GrepTool', 'FileReadTool', 'BashTool']
}

// src/tools/AgentTool/built-in/verificationAgent.ts
export const VERIFICATION_AGENT = {
  agentType: 'verification',
  description: 'Verifies code changes and implementation',
  // 验证代理，专门检查实现是否正确
}
```

### Fork（分叉）：轻量级并行

除了创建全新的子代理，Claude Code 还支持"Fork"——复制当前的对话上下文，让 AI 在一个独立的环境中探索或实验。

```typescript
// Fork 的使用场景
await AgentTool({
  // 不指定 subagent_type，就是 Fork 模式
  description: "Research alternative approaches",
  prompt: "Research 3 different approaches to implement caching..."
  // Fork 会继承当前的完整上下文
})
```

Fork 的特点是：
- **继承上下文**：不需要重新解释背景
- **共享缓存**：与父 Agent 共享提示词缓存，启动更快
- **隔离结果**：Fork 中的尝试不会污染主对话

### 团队（Team）系统

当任务更复杂，需要多个专门的 Agent 协同工作时，可以使用团队功能：

```typescript
// 创建一个团队
await TeamCreateTool({
  name: "project-alpha",
  agents: [
    { type: "frontend-dev", count: 2 },
    { type: "backend-dev", count: 1 },
    { type: "qa", count: 1 }
  ]
})

// 通过 SendMessageTool 给特定 Agent 发送消息
await SendMessageTool({
  to: "frontend-dev-1",
  message: "Implement the login page..."
})
```

### 代理间的通信

子代理之间可以通过 `SendMessageTool` 通信：

```typescript
// Agent A 告诉 Agent B 任务完成
await SendMessageTool({
  to: "backend-dev",
  message: "API specs are ready in /docs/api.md. Start implementing..."
})
```

这种消息传递机制让多个 Agent 能够协调工作，就像一个真正的团队一样。

### 使用代理的最佳实践

从 Agent 的 prompt 中，我们可以总结出一些最佳实践：

```typescript
// 来自 src/tools/AgentTool/prompt.ts
const bestPractices = `
## When to use AgentTool
- For complex, multi-step tasks that can run independently
- When you need parallel work to maximize efficiency
- For tasks requiring specialized tools or knowledge

## When NOT to use AgentTool
- For simple lookups (use GlobTool/GrepTool instead)
- For specific file reads (use FileReadTool directly)
- For tasks requiring the full context of current conversation

## Writing good prompts for agents
- Brief the agent like a colleague who just walked in
- Explain what you're trying to accomplish and why
- Describe what you've already learned or ruled out
- Be specific about scope: what's in, what's out
`
```

## 2.5 Skill 系统：把常用操作变成可复用的技能

### 什么是 Skill？

Skill（技能）是一种**可复用的提示词模板**。你可以把复杂的操作序列定义为一个 Skill，之后只需要说"执行这个 Skill"就能完成整个操作。

这就像手机上的"快捷指令"（iOS Shortcuts）或 IFTTT 自动化——把一系列步骤打包成一个按钮。

### Skill 的结构

Skill 存储在 `~/.claude/skills/` 目录中，以 Markdown 文件的形式存在：

```markdown
---
name: code-review
description: "Review code changes for quality and security"
whenToUse: "When you need to review code changes"
---
# Code Review Skill

## Phase 1: Gather Information
Run `git diff` to see what changed...

## Phase 2: Launch Review
Use the AgentTool to launch a code reviewer...

## Phase 3: Aggregate Results
Wait for the reviewer and summarize findings...
```

Skill 的定义包含：
1. **Frontmatter（元数据）**：名称、描述、使用场景
2. **Prompt（内容）**：具体的执行步骤

### 内置 Skill

Claude Code 自带了一些内置 Skill，保存在 `src/skills/bundled/` 中：

| Skill | 功能 |
|-------|------|
| `simplify` | 代码审查与重构 |
| `remember` | 记住信息供后续使用 |
| `loop` | 循环执行某个命令 |
| `schedule` | 调度远程任务 |
| `verify` | 验证实现是否正确 |

以 `simplify` 为例：

```typescript
// src/skills/bundled/simplify.ts
const SIMPLIFY_PROMPT = `# Simplify: Code Review and Cleanup

Review all changed files for reuse, quality, and efficiency. Fix any issues found.

## Phase 1: Identify Changes
Run \`git diff\` to see what changed...

## Phase 2: Launch Three Review Agents in Parallel
Use the AgentTool to launch all three agents concurrently...

### Agent 1: Code Reuse Review
Search for existing utilities that could replace newly written code...

### Agent 2: Code Quality Review
Review for hacky patterns: redundant state, parameter sprawl...

### Agent 3: Efficiency Review
Review for unnecessary work, missed concurrency...

## Phase 3: Fix Issues
Wait for all three agents, aggregate findings, fix issues...
`
```

这个 Skill 非常精妙——它不是简单地"审查代码"，而是定义了三个阶段，每个阶段有明确的目标和方法。

### Skill 的执行

当用户说 `/simplify` 时，SkillTool 会：

```typescript
// 简化版流程
async function executeSkill(skillName: string) {
  // 1. 加载 Skill 定义
  const skill = await loadSkill(skillName)

  // 2. 获取 Skill 的 prompt
  const prompt = await skill.getPromptForCommand(args)

  // 3. 执行 Skill（可能是启动子代理，也可能是直接执行）
  await executePrompt(prompt)
}
```

### Skill 与 Agent 的关系

Skill 和 Agent 都是代码复用的方式，但适用场景不同：

| 特性 | Skill | Agent |
|------|-------|-------|
| **本质** | 提示词模板 | 独立运行的进程 |
| **执行方式** | 在当前上下文中执行 | 可在后台独立运行 |
| **适用场景** | 标准化的工作流程 | 需要专门工具或长时间运行 |
| **示例** | `/simplify` 代码审查流程 | `exploreAgent` 专门的文件搜索 |

### 自定义 Skill

用户可以创建自己的 Skill。只需要在 `~/.claude/skills/` 目录添加一个 Markdown 文件：

```markdown
---
name: my-workflow
description: "My custom workflow"
whenToUse: "When I need to do X"
---
# My Custom Workflow

Step 1: ...
Step 2: ...
Step 3: ...
```

这使得团队可以共享和标准化他们的工作流程。

## 2.6 插件系统：像手机应用一样的扩展能力

### 为什么需要插件系统？

Skill 让你可以复用自己团队的工作流程。但如果你想让全世界的人都来使用你的成果呢？这就需要插件系统。

插件系统让第三方开发者可以为 Claude Code 添加新功能，就像手机应用让智能手机变得无所不能一样。

### 插件的结构

Claude Code 的插件保存在 `~/.claude/plugins/` 目录中：

```
~/.claude/plugins/
└── my-plugin/
    ├── plugin.json      # 插件元数据
    ├── skills/          # 插件提供的 Skill
    ├── agents/          # 插件提供的 Agent
    └── index.ts         # 插件入口
```

### 插件的注册

插件通过 `plugin.json` 来声明自己的身份和提供的功能：

```json
{
  "name": "my-plugin",
  "version": "1.0.0",
  "description": "A useful plugin for...",
  "skills": [
    {
      "name": "my-skill",
      "path": "./skills/my-skill.md"
    }
  ],
  "agents": [
    {
      "type": "my-agent",
      "description": "Specialized agent for...",
      "prompt": "You are a helpful..."
    }
  ]
}
```

### 插件的加载

Claude Code 在启动时会扫描并加载所有插件：

```typescript
// src/utils/plugins/pluginLoader.ts（简化）
async function loadPlugins() {
  const pluginDirs = findPluginDirs()

  for (const dir of pluginDirs) {
    const plugin = await loadPlugin(dir)
    registerPlugin(plugin)
  }
}
```

### 插件提供的功能

插件可以扩展 Claude Code 的多个方面：

| 扩展点 | 说明 | 示例 |
|--------|------|------|
| **Skill** | 添加新的可复用技能 | 提供特定框架的模板 |
| **Agent** | 添加新的专门化 Agent | 提供代码审查、文档生成等 |
| **Tool** | 添加新的工具能力 | 连接第三方 API |
| **Command** | 添加新的 slash 命令 | `/jira`、`/linear` |
| **Hook** | 在特定时机执行自定义逻辑 | 提交前运行测试 |

### 内置插件 vs 用户插件

Claude Code 的插件系统设计了一个有趣的边界：

```typescript
// src/plugins/bundled/index.ts
/**
 * Built-in Plugin Initialization
 *
 * Initializes built-in plugins that ship with the CLI...
 */
export function initBuiltinPlugins(): void {
  // 内置插件在这里注册
  // 用户插件从 ~/.claude/plugins/ 加载
}
```

这意味着：
- **内置插件**：随 Claude Code 一起发布，用户不能禁用
- **用户插件**：用户可以自由安装、启用、禁用

### MCP：Model Context Protocol

最后要提到的是 MCP（Model Context Protocol）。这是 Claude Code 与外部工具和服务集成的一种标准化方式：

```typescript
// MCP 服务器配置示例
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "~/projects"]
    },
    "github": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-github"],
      "env": {
        "GITHUB_TOKEN": "xxx"
      }
    }
  }
}
```

通过 MCP，Claude Code 可以访问任何实现了 MCP 协议的工具——就像 USB-C 端口让任何符合标准的设备都能连接一样。

### 小结：扩展性的设计哲学

插件系统的设计体现了可扩展性的核心原则：

1. **开放架构**：任何人都可以添加新功能
2. **标准化接口**：通过 MCP 等协议确保互操作性
3. **沙箱隔离**：插件在受限的环境中运行，不会危及系统安全
4. **用户控制**：用户可以完全控制安装和卸载

## 本章小结

在这一章中，我们深入探讨了 Claude Code 的几个核心设计理念：

| 系统 | 核心思想 | 类比 |
|------|----------|------|

| **技能系统** | 模板复用、标准化流程 | 快捷指令 |
| **插件系统** | 开放扩展、生态建设 | 应用商店 |

这些系统相互配合，让 Claude Code 不仅仅是一个"能聊天的 AI"，而是一个真正能够**帮你完成复杂软件工程任务**的智能助手。
