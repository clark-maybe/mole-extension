## 3.1 查询引擎（QueryEngine）：AI 是怎么调用的？

### QueryEngine 是什么？

在 Claude Code 中，所有与 AI 的对话都通过 `QueryEngine` 来处理。你可以把它想象成整个系统的心脏——每一次 AI 回复、每一个工具调用，都经过这里。

```typescript
// src/QueryEngine.ts
export class QueryEngine {
  private config: QueryEngineConfig
  private mutableMessages: Message[]
  private abortController: AbortController
  private permissionDenials: SDKPermissionDenial[]
  private totalUsage: NonNullableUsage
  // ...
}
```

### 核心配置

创建 QueryEngine 时需要传入一堆配置（`QueryEngineConfig`），这就像组装一台电脑——不同的配置会得到不同能力的 AI：

```typescript
export type QueryEngineConfig = {
  cwd: string                        // 当前工作目录
  tools: Tools                      // 可用的工具列表
  commands: Command[]                // 可用的命令列表
  mcpClients: MCPServerConnection[]  // MCP 服务器连接
  agents: AgentDefinition[]          // Agent 定义
  canUseTool: CanUseToolFn          // 权限检查函数
  getAppState: () => AppState       // 应用状态获取函数
  // ...
  customSystemPrompt?: string        // 自定义系统提示词
  userSpecifiedModel?: string       // 指定使用的模型
  thinkingConfig?: ThinkingConfig    // 思考模式配置
  maxTurns?: number                // 最大对话轮次
  maxBudgetUsd?: number             // 最大花费限制
}
```

### 主循环：消息处理

QueryEngine 的核心是一个"消息处理循环"：

```typescript
// 简化版流程
async function submitMessage(userMessage: Message): Promise<void> {
  // 1. 将用户消息添加到历史
  this.mutableMessages.push(userMessage)

  // 2. 构建 AI 请求
  const apiRequest = await this.buildAPIRequest()

  // 3. 发送给 AI
  const response = await this.callAI(apiRequest)

  // 4. 处理 AI 的响应
  for (const block of response.content) {
    if (block.type === 'text') {
      // 处理文本回复
      this.handleTextBlock(block)
    } else if (block.type === 'tool_use') {
      // 处理工具调用
      await this.handleToolCall(block)
    }
  }

  // 5. 将 AI 回复添加到历史
  this.mutableMessages.push(aiResponse)
}
```

### 工具调用的处理

当 AI 说"我要调用工具"时，QueryEngine 会：

```typescript
// 处理工具调用的简化流程
async function handleToolCall(toolUse: ToolUseBlock) {
  // 1. 检查权限
  const permission = await this.checkPermission(toolUse.name)

  if (!permission.allowed) {
    // 被拒绝，返回错误给 AI
    return this.createToolError(toolUse.id, permission.reason)
  }

  // 2. 执行工具
  const result = await this.executeTool(toolUse.name, toolUse.input)

  // 3. 返回结果给 AI
  return this.createToolResult(toolUse.id, result)
}
```

### 重试与错误处理

AI 调用不是 100% 可靠的——网络会抖动，API 会限流。QueryEngine 有一套完善的错误处理机制：

```typescript
// 错误分类处理
const categorizeRetryableAPIError = (error: Error) => {
  if (error instanceof RateLimitError) {
    return 'rate_limit'  // 限流：等待后重试
  }
  if (error instanceof NetworkError) {
    return 'network'     // 网络错误：等待后重试
  }
  if (error instanceof ServerError) {
    return 'server'      // 服务器错误：等待后重试
  }
  if (error instanceof AuthenticationError) {
    return 'auth'        // 认证错误：不重试，需要修复
  }
  return 'unknown'       // 其他错误：决定是否重试
}
```

### 重连与错误恢复

网络不稳定是常态，桥接系统有完善的重连机制：

```typescript
// 重连配置
const DEFAULT_BACKOFF: BackoffConfig = {
  connInitialMs: 2_000,     // 初始重连间隔 2 秒
  connCapMs: 120_000,       // 最大重连间隔 2 分钟
  connGiveUpMs: 600_000,    // 10 分钟后放弃
  generalInitialMs: 500,
  generalCapMs: 30_000,
  generalGiveUpMs: 600_000,
}

// 重连逻辑
async function withRetry<T>(operation: () => Promise<T>): Promise<T> {
  let attempt = 0
  while (true) {
    try {
      return await operation()
    } catch (error) {
      if (isRetryable(error) && attempt < maxAttempts) {
        attempt++
        await sleep(calculateBackoff(attempt))
        continue
      }
      throw error
    }
  }
}
```

```typescript
// 定时刷新
function scheduleTokenRefresh(expiresIn: number): void {
  // 在令牌过期前一段时间就刷新

  setTimeout(refreshToken, refreshTime)

```

```typescript
// src/services/mcp/client.ts
async function connectToServer(config: McpServerConfig): Promise<MCPServerConnection> {
  // 1. 启动 MCP 进程
  const process = spawn(config.command, config.args)

  // 2. 初始化握手
  const { tools, resources } = await initializeProtocol(process)

  // 3. 注册工具
  for (const tool of tools) {
    registerTool({
      name: tool.name,
      description: tool.description,
      inputSchema: tool.inputSchema,
      handler: (input) => callMCPProcess(process, tool.name, input)
    })
  }

  return { process, tools, resources }
}
```

## 3.4 启动优化：并行预加载让启动更快

### 启动速度的重要性

对于一个 CLI 工具来说，启动速度至关重要。没有人愿意等好几秒才能开始工作。Claude Code 的 main.tsx 只有约 80 万字节，但加载的模块却非常复杂。

### 预加载策略

Claude Code 在真正开始加载代码之前，就已经开始准备数据了：

## 3.5 条件编译：只编译需要的功能（feature flags）

### 什么是 feature flags？

Feature flags（功能开关）是一种强大的技术，它允许你在不发布不同版本的情况下，动态启用或禁用功能。

Claude Code 使用 Bun 内置的 `feature()` 函数来实现这个功能：

```typescript
import { feature } from 'bun:bundle'

// 如果 VOICE_MODE 功能未启用，这段代码就不会被包含在最终产物中
if (feature('VOICE_MODE')) {
  const voiceModule = require('./commands/voice/index.js')
  // ...
}
```

### 工作原理

Bun 的 `feature()` 函数在构建时会根据环境变量来决定包含哪些代码：

```bash
# 构建时指定启用的功能
bun build --target node --feature VOICE_MODE --feature KAIROS src/main.tsx
```

这就像乐高套装中的"可选包"——你只拿到你购买的那些部件。

### Feature 列表

Claude Code 定义了大量的 feature flags：

| Feature | 说明 |
|---------|------|
| `PROACTIVE` | 主动模式 |
| `KAIROS` | KAIROS 助手模式 |
| `BRIDGE_MODE` | IDE 桥接模式 |
| `DAEMON` | 守护进程模式 |
| `VOICE_MODE` | 语音模式 |
| `AGENT_TRIGGERS` | 定时代理触发 |
| `MONITOR_TOOL` | 监控工具 |
| `COORDINATOR_MODE` | 协调器模式 |
| `HISTORY_SNIP` | 历史截断 |
| `EXPERIMENTAL_SKILL_SEARCH` | 实验性技能搜索 |

### 代码示例

Feature flags 的使用非常普遍：

```typescript
// src/commands.ts
const proactive =
  feature('PROACTIVE') || feature('KAIROS')
    ? require('./commands/proactive.js').default
    : null

const voiceCommand = feature('VOICE_MODE')
  ? require('./commands/voice/index.js').default
    : null

const bridge = feature('BRIDGE_MODE')
  ? require('./commands/bridge/index.js').default
    : null
```

### 条件编译的优势

这种技术带来了几个显著的优势：

1. **减小包体积**：不需要的功能不会被打包
2. **灰度发布**：可以逐步启用新功能
3. **A/B 测试**：可以同时测试不同版本的实现
4. **环境适配**：不同版本（内部版、企业版）可以有不同功能

### 与 TypeScript 的配合

虽然这不是严格的条件编译，但 TypeScript 的类型系统也为代码的条件化提供了支持：

```typescript
// 类型层面
type FeatureFlags = {
  VOICE_MODE: boolean
  KAIROS: boolean
  // ...
}

// 类型化的特性检查
function checkFeature(flag: keyof FeatureFlags): boolean {
  return feature(flag)
}
```


### 目录组织的哲学

Claude Code 的目录组织遵循了几个原则：

1. **按功能分**：所有 API 相关代码在 `services/api/`
2. **按领域分**：所有 MCP 相关代码在 `services/mcp/`
3. **按层次分**：`commands/` 是命令，`tools/` 是工具，`components/` 是 UI

## 本章小结

在这一章中，我们深入到了 Claude Code 的实现细节：

| 主题 | 核心要点 |
|------|----------|
| **QueryEngine** | AI 调用的核心，处理消息循环、工具调用、错误重试、思考模式 |
| **桥接系统** | IDE 与 CLI 通信的桥梁，使用 WebSocket + JSON 协议，有完善的重新连接机制 |
| **服务层** | 分为 API、MCP、OAuth、分析等多个服务，各司其职 |
| **启动优化** | 并行预加载关键资源、延迟加载重型模块、缓存复用 |
| **条件编译** | 通过 `feature()` 实现"一码多构"，减小包体积，支持灰度发布 |

这些技术的组合，让 Claude Code 能够在保持功能丰富的同时，保持良好的性能和启动速度。
